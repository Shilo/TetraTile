### Composed-Canvas Visual Regression Test (Task 5.2)

`addons/penta_tile/tests/pixellab_visual_regression_test.gd` (~250 LOC):

- **For each PIXLAB layout** (top-down + side-scroller):
  1. Loads sample PNG as `bitmask_template`
  2. Builds `TileSet` via `layout.get_fallback_tile_set()` (8×8 grid at 16px tiles derived from 128/8 PNG dimensions)
  3. Configures fresh `PentaTileMapLayer`, paints mixed pattern (3×3 rect + isolated cell + horizontal line + L-shape — varied corner-mask coverage)
  4. Composes virtual canvas via `Image.create()` + per-cell `blit_rect(_apply_transform(src_tile, transform))` (transform=0 always per D-90, but the helper preserves the analog's structure)
  5. Asserts:
     - **(a) Every painted cell renders** (no atlas out-of-bounds dispatch)
     - **(b) Canvas size matches painted region's pixel bounds**
     - **(c) Bit-stability — two consecutive `rebuild()` calls produce identical canvas data** (D-89 first-cell pick must be deterministic; PackedByteArray `==` element-wise compare)
- **save_png** on every run to `user://pixellab_visual_regression_<layout>.png` for visual inspection (Methodology #4)

## Verify-the-Regression Cycle (Test Methodology #5)

**Mandatory cycle confirmed.** Stashed `mask_to_atlas` body in `penta_tile_layout_pixel_lab_top_down.gd` to force every cell to dispatch to `Vector2i(0, 0)` (the top-left atlas cell — solid grey). Reran the test and inspected the saved canvas.

| State | Test result | Saved canvas appearance |
| --- | --- | --- |
| Working (HEAD) | ALL PASS | Varied tile content per mask — proper edges, corners, fills |
| Stashed `mask_to_atlas → (0, 0)` | ALL PASS (assertions don't catch deterministic-but-broken dispatch) | Uniform grey tile across entire painted region — obviously broken to the eye |

**Why bbox/non-empty/bit-stability assertions don't catch this stash:** `Vector2i(0, 0)` IS a valid atlas coord and IS deterministic. Bit-stability holds (rebuild produces identical canvas). Canvas size matches painted bounds. Opaque pixels exist (the (0,0) tile is solid). The catch mechanism for this regression class is the saved PNG — eyeball verification per Test Methodology #4.

This is **expected and documented** in the plan's action steps: "the bit-stability assertion still passes since the dispatch is deterministic, but the canvas-rendered output will look obviously broken — eyeball the saved `user://pixellab_visual_regression_PixelLabTopDown.png`." A fully-stricter test would require a per-mask expected first-cell coord assertion — that contract is already locked by Plan 04's `pixellab_first_cell_test`. This visual-regression test's job is the orthogonal "do the layouts render against a real PixelLab atlas without obvious breakage" question, with the saved canvas as the definitive artifact.

After visual confirmation that the broken canvas was visibly different from the working canvas, the layout file was restored and the test re-run to confirm `ALL PASS`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] PackedByteArray.hash() does not exist in Godot 4.6**

- **Found during:** Task 5.2 first test run
- **Issue:** Plan action's reference body called `canvas_a.get_data().hash() != canvas_b.get_data().hash()` but Godot 4.6's `PackedByteArray` has no `hash()` method. Parse error: `Cannot find member "hash" in base "PackedByteArray".`
- **Fix:** Replaced with direct `PackedByteArray` equality (`canvas_a.get_data() != canvas_b.get_data()`). PackedByteArray supports element-wise `==` in GDScript 2 — sufficient for bit-stability comparison and arguably more correct (content-based equality vs. hash collision risk).
- **Files modified:** `addons/penta_tile/tests/pixellab_visual_regression_test.gd` (line 145)
- **Commit:** `aff6f35` (folded into Task 5.2 commit; documented in commit message)

No Rule 2/3/4 deviations.
