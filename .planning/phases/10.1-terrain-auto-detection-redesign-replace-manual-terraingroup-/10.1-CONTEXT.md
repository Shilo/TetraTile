# Phase 10.1: Terrain Auto-Detection Redesign - Context

**Gathered:** 2026-05-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the Phase 10 `PentaTileTerrainGroup` Resource + per-terrain layout array with auto-detection: terrain count derived from `atlas_grid_size.y` (each atlas row = one terrain), Godot native `TerrainSets` for name/color storage (not solving), single shared `layout: PentaTileLayout` for all terrains. Spike 009 validated all 6 auto-detection behaviors against Godot 4.6.2. ~280 LOC net savings (~400 removed, ~120 added). Excludes new feature work outside terrain identity — variation, source_id, Slope, and passthrough survive unchanged.

Explicitly NOT in scope: new variation features, new layout subclasses, editor tooling, VirtuMap passthrough wiring changes (passthrough API stays, just terrain identity resolution changes internally).
</domain>

<decisions>
## Implementation Decisions

### Scope Boundary — What Survives vs Gets Deleted
- **D-01:** DELETE: `PentaTileTerrainGroup` class (`addons/penta_tile/layouts/penta_tile_terrain_group.gd` + `.uid`), `terrain_group: PentaTileTerrainGroup` export on `PentaTileMapLayer`, terrain_group setter, `_build_terrain_index()`, `_terrain_index` Dictionary, `_on_terrain_group_changed()` signal handler, all terrain_group fallback routing paths, per-terrain Penta synthesis bridge in `_ensure_visual_layers` (no multi-layout array to iterate).
- **D-02:** KEEP unchanged from Phase 10: `variation_mode` enum (SINGLE/PROBABILITY/STRIP), `source_id` on `PentaTileAtlasSlot`, `terrain_mode()` virtual on `PentaTileLayout` base + all 9 subclass overrides, `compute_mask(strip_index: int = 0)` signature, `set_cell_passthrough()` method, `PentaTileLayoutSlope` subclass.
- **D-03:** Fallback routing returns to v0.2.0 behavior — when `tile_set == null`, routes through `layout.get_fallback_tile_set()`. No terrain-aware fallback path (deleted with TerrainGroup).
- **D-04:** Auto-detection triggers: `tile_set` setter AND `tile_set.changed` signal (via deferred coalescer). Does NOT trigger in `_ready()` — the setter fires during scene deserialization and covers the initial-load case.

### Dual-Grid Terrain Boundary Dispatch
- **D-05:** Per-corner single-shared-layout dispatch replaces Phase 10's per-terrain-layout dispatch. Each dual-grid display cell's 4 corners get terrain IDs from their 4 logic cell neighbors. Each corner dispatches through the SHARED layout's `mask_to_atlas` with `strip_index = terrain_id`. The `sample_fn` for mask computation filters: only same-terrain neighbors count as filled; other-terrain neighbors count as empty. This naturally creates boundaries — each terrain's tiles self-edge at terrain borders.
- **D-06:** `terrain_precedence: Array[int]` deleted (was on TerrainGroup). No paint-order stacking — each corner renders once, no overlaps to sort.
- **D-07:** `terrain_mode()` virtual reports the Godot `TerrainMode` for terrain_set 0 creation (MATCH_CORNERS for Penta/DualGrid16/Wang2Corner/PixelLab, MATCH_SIDES for Wang2Edge/Min3x3, MATCH_CORNERS_AND_SIDES for Blob47Godot). Used when creating terrain_set 0. PentaTile never reads Godot peering bits — this is storage-only mode information.

### Variation (Preserved from Phase 10)
- **D-08:** Variation pools indexed per-terrain via terrain index (not TerrainGroup). `mask_to_atlas` scans the atlas row for tiles sharing the same mask config, pools them by `TileData.probability` weight.
- **D-09:** Deterministic hash shape preserved: `hash(Vector4i(cell.x, cell.y, terrain_id, _global_variation_seed))`. Rebuild produces identical variation picks.
- **D-10:** STRIP mode preserved — picks from terrain's atlas row (same atlas_coords.y = terrain_id).

### Auto-Detection UX
- **D-11:** Fully automatic — no inspector button, no manual trigger. `_auto_detect_terrains()` runs in `tile_set` setter + on `tile_set.changed`. Creates `terrain_set 0` with MATCH_CORNERS mode (covers 6/9 layouts as correct default). User opens Godot's built-in TileSet bottom panel → Terrain Sets tab to rename/recolor.
- **D-12:** Terrains created for ALL `atlas_grid_size.y` rows (including empty rows — artist may add tiles later). `TileData.terrain = atlas_coords.y` for every tile.
- **D-13:** `_resolve_terrain_id(cell)` reads `TileData.terrain` directly from `source.get_tile_data(atlas_coords, 0).terrain`. No source-offset map — `TileData.terrain` is set to the sequential terrain ID during auto-detection.
- **D-14:** Names auto-generated as `"Terrain N"` with deterministic-hash colors. User customizations survive auto-rebuild via the spike 009 snapshot-before-clear pattern: before clearing terrain_set 0, snapshot `name[id]` and `color[id]`; after recreating, restore names that don't start with `"Terrain "` (user-set). Colors with non-hash matches also restored.
- **D-15:** Multiple atlas sources stack terrain IDs sequentially (source 0 rows 0..R0-1 → terrains 0..R0-1, source 1 rows 0..R1-1 → terrains R0..R0+R1-1). `TileData.terrain` holds the sequential ID.

### Test Strategy
- **D-16:** 4 new tests ship:
  1. `terrain_autodetect_test.gd` — validates `_auto_detect_terrains()` creates correct terrain set from atlas grid dimensions, names/colors, `TileData.terrain = atlas_coords.y` assignments, snapshot-preserve pattern
  2. `terrain_dispatch_test.gd` — 8 layouts × 8 key patterns × 2-terrain boundaries (clean transitions at terrain edges). Patterns: L_shape, T_shape, plus_shape, 2×2_bicolor, line_h_int, hollow_ring, 1×1, 3×3
  3. `terrain_determinism_test.gd` — rebuild produces bit-identical output across runs (hash-based)
  4. `terrain_sample_terrains_test.gd` — loads `sample_terrains.png` (160×96, 32px tile grid → auto-detects 3 terrains), paints multi-terrain patterns on real artwork, validates boundary cells render correct terrain tiles
- **D-17:** 5 old TerrainGroup-coupled tests deleted: `terrain_fallback_test.gd`, `terrain_integration_test.gd`, `terrain_index_test.gd`, `single_grid_cross_terrain_test.gd`, `dual_grid_terrain_test.gd`. `run_tests.ps1` removes their entries before adding the 4 new ones.

### the agent's Discretion
- Exact `_auto_detect_terrains()` function structure and internal helper decomposition
- Snapshot-before-clear implementation details (exact hash function for colors, detection heuristic for "user-set" vs "auto-generated" names)
- `_resolve_terrain_id()` implementation details and null-safety guards
- Per-corner sample_fn construction for dual-grid same-terrain filtering
- Test fixture construction (programmatic TileSet + Image creation per established test patterns)
- Error/warning behavior when tile_set has no atlas sources or zero tiles
</decisions>

## Existing Code Insights

### Reusable Assets
- **`PentaTileMapLayer._update_cells()`** — Main autotile egress point. Terrain dispatch extends `_paint_via_layout()` and `_paint_dual_grid()` within this pipe. Hot-path addition: ~15 lines for terrain resolution reading `TileData.terrain`.
- **`PentaTileLayoutPenta` AUTO_STRIP** — Existing `resolve_display_strip()` and per-strip dispatch. Per-terrain Penta strips naturally map to `strip_index = terrain_id`.
- **`PentaTileLayout.terrain_mode()`** — Already implemented across all 9 subclasses from Phase 10. Reused for terrain_set 0 creation mode selection.
- **`PentaTileLayout.compute_mask(coord, sample_fn, strip_index=0)`** — Already has strip_index parameter from Phase 10 D-09. Reused for terrain-aware mask filtering.
- **`PentaTileAtlasSlot.source_id`** — Already has source_id field from Phase 10. Unchanged.
- **`rand_weighted()` pattern** — Already used in v0.2.0 for Penta synthesis and Phase 10 variation. Repurposed with terrain_id in hash seed.

### Established Patterns
- **`_queue_rebuild` deferred coalescer** — Auto-detection rides the same deferred-rebuild path. `tile_set.changed` → queued → `_auto_detect_terrains()` → `_queue_rebuild()`. No new queue mechanism.
- **Deferred signal handling** — `_auto_detect_terrains()` follows the existing `_on_layout_changed` pattern: receive signal, defer work, coalesce.
- **Custom data layers** — Existing `penta_role` and `penta_lock_rotation` layers establish the custom-data-for-metadata pattern. Phase 10.1 does NOT add a new custom data layer (terrain identity is in TileData.terrain, not custom data).
- **Deterministic hashing** — All variation uses `RandomNumberGenerator.seed = hash(Vector4i(...))` per PITFALLS §2. Never `randi()`.
- **Composed-canvas tests** — Established in Phase 2 UAT. All terrain tests use blit-and-assert pattern.

### Integration Points
- **`tile_set` setter + `tile_set.changed` signal** — Both trigger `_auto_detect_terrains()`. The tile_set setter already exists; the changed signal handler is new.
- **`_resolve_source_id()`** — Currently returns global source. Already gains terrain-aware routing when `PentaTileAtlasSlot.source_id >= 0` (preserved from Phase 10).
- **`_resolve_terrain_id()`** — Simplified: reads `TileData.terrain` directly instead of custom data ± TerrainGroup lookup.
- **`get_fallback_tile_set()`** — Returns to v0.2.0 path (layout only, no terrain_group override).

<specifics>
## Specific Ideas

- Auto-detection should be invisible — assign a TileSet with atlas rows and terrains exist. No manual configuration.
- User renames/recolors terrains via Godot's native TerrainSets inspector panel (TileSet bottom panel → Terrain Sets tab). Customizations survive auto-rebuild.
- `terrain_set 0` uses MATCH_CORNERS as default mode (covers 6/9 layouts). Wang2Edge/Min3x3 users who need MATCH_SIDES can manually adjust in the inspector.
- Tests must verify cross-terrain boundaries visually — two terrains side-by-side should produce clean edge tiles on both sides.
- The `sample_terrains.png` fixture test is the primary "does this work with real art" verification.
</specifics>
