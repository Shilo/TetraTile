## Gate 1 Resolution: PENTA-SYNTH-05 Sub-Region Anchoring (D-69 LOCKED)

User declined Spike 004; this plan locks the anchoring spec.

### Decision: Slot 0 is treated as a 2×2 sub-grid for synthesis

Slot 0 (`IsolatedCell`) visually IS four outer corners + four edges + a center fill packed into one tile. We exploit that geometry: the synthesizer reads four 1/2-tile sub-rects from slot 0's pixels and assembles them (with rotation) into the synthesized archetypes. This matches Penta's existing dual-grid math where each display cell already corresponds to a 2×2 quadrant of logic cells.

### Diagram (ONE mode — slot 0 is the only authored tile)

```
        SLOT 0 (IsolatedCell) — single tile_size×tile_size source
        ────────────────────────────────────
        | TL_quad  |  TR_quad |
        |   = OC0  |   = OC1  |
        | (rotates |(rotates  |
        |  to TL,  | to TR,   |
        |  BL, BR  | BL, BR   |
        |  via     | via      |
        |  TRANSFORM)| TRANSFORM)|
        |──────────┼──────────|
        | BL_quad  |  BR_quad |
        |   = OC2  |   = OC3  |
        ────────────────────────────────────

        Each quadrant is tile_size/2 × tile_size/2 pixels.
        Sub-region origins (top-left corner, in source pixels):
          TL_quad  → (0,             0)
          TR_quad  → (tile_size/2,   0)
          BL_quad  → (0,             tile_size/2)
          BR_quad  → (tile_size/2,   tile_size/2)
```

### Archetype derivation from slot 0 (LOCKED for ALL modes — TWO/THREE/FOUR also use this when their respective slot is unfilled)

| Synthesized archetype | Source sub-rect of slot 0 | Transform | Output sub-region |
|---|---|---|---|
| **OuterCorner** | NO sub-region extraction — slot 0 (IsolatedCell) used verbatim | TRANSFORM_TRANSPOSE/FLIP_H/FLIP_V combinations applied at render time via `_pack_alternative` for the 4 corner orientations | NO synthesized output cell — mask_to_atlas points at slot 0 with rotation flags |
| **Fill** | center 50% of slot 0 (rect spanning x=tile_size/4..3×tile_size/4, y=tile_size/4..3×tile_size/4), STRETCHED to tile_size×tile_size | none | full tile |
| **Border** | half-slab of slot 0 — rect (0, tile_size/2)..(tile_size, tile_size) (the "S" edge slab) STRETCHED to tile_size×tile_size | rotated via TRANSFORM flags for N/E/S/W variants | full tile, four rotation reuses |
| **InnerCorner** | three-quadrant region of slot 0 — rect (tile_size/2, 0)..(tile_size, tile_size) UNION (0, tile_size/2)..(tile_size, tile_size) i.e. "L-shape" with TR quadrant cut out, STRETCHED to tile_size×tile_size | rotated for the four corner variants | full tile, four rotation reuses |
| **OppositeCorners** | concatenation of TL_quad + BR_quad rendered into a fresh tile_size×tile_size canvas at TL + BR positions, transparent elsewhere | none for mask 9, TRANSFORM_FLIP_H for mask 6 | full tile, two rotation reuses |

**OuterCorner exception (Path B locked):** Unlike Fill/Border/InnerCorner/OppositeCorners (which DO produce dedicated synthesized output cells), OuterCorner has NO synthesized slot. The synthesizer skips it entirely. Wave 3's `mask_to_atlas` returns `_SLOT_ISOLATED_CELL` (slot 0) with `_ROTATE_0`/`_ROTATE_90`/`_ROTATE_180`/`_ROTATE_270` for masks 4/1/2/8 respectively. This is the v0.1 Tetra4 rotation-reuse precedent, applied to slot 0 in the synthesized atlas. Visual fidelity tradeoff: rotating an entire IsolatedCell tile shows the IsolatedCell's other 3 corners + edges + center fill alongside the "intended" corner; this is acceptable per the "sacrificing quality for less quantity" prototyping intent. Artists wanting pixel-perfect OuterCorner rendering must author their IsolatedCell PNG with appropriately faded extras (transparent or near-transparent center/edges) so the rotated rendering reads as a single corner.

### Tile-size constraints (LOCKED)

1. **Square tiles only.** `tile_size.x == tile_size.y`. If non-square, `_synthesize_strip` calls `push_error("PentaTileLayoutPenta: source tile_set requires square tiles for synthesis; got <X>×<Y>")` and returns null TileSet (caller renders nothing + logs a `update_configuration_warnings()` entry).
2. **Even dimensions.** `tile_size.x % 2 == 0` (so half-tile sub-rects are integer). Recommended minimum 4 px (so quadrants are at least 2×2 px each). Below 4 px synthesis still proceeds but emits a configuration warning ("Tile size below 4 px; synthesized archetypes will be visually crude").
3. **Power-of-2 strongly recommended** (4/8/16/32/64) but not enforced (just a warning).

### Degradation behavior on non-square / mid-tier-mode atlases

- **Non-square source tiles:** synthesis is REFUSED (returns null TileSet + warning). Artist must square their atlas. This is a documented limitation, NOT a silent fallback.
- **TWO/THREE/FOUR mode** (slots 1+ authored, slot 0 still feeds the missing archetypes): the same anchoring math above applies; slot 0 is still the sub-region source for whichever archetypes are NOT covered by an authored slot. Fill in TWO mode comes from slot 1 directly (no synthesis); InnerCorner in THREE mode synthesizes from slot 0 even though Border/Fill are now authored. **Synthesis is per-archetype, not per-mode** — the mode only determines WHICH archetypes are synthesized vs read directly.
- **Slot 0 is REQUIRED in all modes.** If slot 0 is empty (e.g., AUTO_STRIP detects a strip starting at slot 1), `update_configuration_warnings()` fires the AUTO_STRIP-gap warning per PENTA-SYNTH-08 and the strip is rendered empty (graceful degradation, not crash).

### Why this anchoring choice

The 2×2-quadrant interpretation of slot 0 has the practical property that an artist authoring an IsolatedCell tile naturally draws four outer corners + four edges + a fill — Penta's archetype set is geometrically embedded in the IsolatedCell silhouette. The only non-obvious step is the "stretch" used for Border/Fill/InnerCorner (which extracts a sub-rect and scales it back to tile_size); this is acceptable because (a) it's a graceful approximation of artist intent in the absence of dedicated slots, and (b) the entire synthesis pipeline is intentionally a fast prototyping path — artists who care about pixel-perfect visuals upgrade to FIVE mode (all archetypes hand-drawn). This is the "sacrificing quality for less quantity" intent locked in CONTEXT.md `<specifics>`.

The alternative (axis-aligned slabs without stretching, packing four reduced-size sub-archetypes inside one synthesized tile) was considered and rejected because it produces a tile_size/2 effective resolution which breaks downstream grids (other Penta cells expect tile_size).

## Gate 2 Resolution: PENTA-SYNTH-06 Polygon Transform Math (D-70 LOCKED)

### Local-origin convention (LOCKED): tile-CENTER

All polygon vertex math operates in tile-center-local coordinates: `(0, 0)` is the geometric center of a tile_size×tile_size tile; vertices range (-tile_size/2, -tile_size/2) to (+tile_size/2, +tile_size/2).

Why tile-center: (a) flip transforms become simple sign negation `(-x, y)` / `(x, -y)`; (b) transpose becomes `(y, x)` with no offset adjustment; (c) matches Godot's convention for `TileData.collision_polygon_n.points` which are stored relative to tile center. PITFALLS.md §1 confirms `TRANSFORM_FLIP_H = 4096`, `TRANSFORM_FLIP_V = 8192`, `TRANSFORM_TRANSPOSE = 16384`.

### Per-flag transformation formulas (LOCKED)

For each vertex `v = Vector2(x, y)` in tile-center-local coords:

```
TRANSFORM_FLIP_H (4096):     v' = Vector2(-x,  y)       # mirror across vertical axis
TRANSFORM_FLIP_V (8192):     v' = Vector2( x, -y)       # mirror across horizontal axis
TRANSFORM_TRANSPOSE (16384): v' = Vector2( y,  x)       # swap x/y (90° rotation around origin combined with one flip)
```

When multiple flags combine, apply in this canonical order: TRANSPOSE first, then FLIP_H, then FLIP_V. (This matches Godot's `TileData` rendering order — the engine applies transpose before flips internally.)

### Worked example — vertex (0.25, 0.75) under all 8 transform combinations

Using `tile_size = 16`, vertex `v = Vector2(0.25, 0.75)` in tile-center-local coords. The 8 combinations (3 flag bits = 2³ outputs):

| Flags (binary `TFFV`) | TRANSPOSE | FLIP_H | FLIP_V | Output v' |
|---|---|---|---|---|
| 0 (identity) | no | no | no | (0.25, 0.75) |
| 4096 (FLIP_H) | no | yes | no | (-0.25, 0.75) |
| 8192 (FLIP_V) | no | no | yes | (0.25, -0.75) |
| 12288 (FLIP_H + FLIP_V) | no | yes | yes | (-0.25, -0.75) |
| 16384 (TRANSPOSE) | yes | no | no | (0.75, 0.25) |
| 20480 (TRANSPOSE + FLIP_H) | yes | yes | no | (-0.75, 0.25) |
| 24576 (TRANSPOSE + FLIP_V) | yes | no | yes | (0.75, -0.25) |
| 28672 (TRANSPOSE + FLIP_H + FLIP_V) | yes | yes | yes | (-0.75, -0.25) |

(Apply TRANSPOSE first → swap to (0.75, 0.25), then FLIP_H/V on that result.)

### Sub-region polygon clipping (LOCKED)

For ONE/TWO/THREE modes where the synthesized tile uses only a sub-rect of slot 0's polygon area: use **axis-aligned-rectangle polygon clipping** (not full Sutherland-Hodgman). Sub-regions are always axis-aligned rectangles per Gate 1, so the simpler algorithm suffices and is significantly cheaper.

Algorithm (per source polygon):
```
for each polygon p in source_tile.collision_polygons:
    clipped = []
    for each vertex v_i in p.points:
        v_i_in_subrect = sub_rect.has_point(v_i)
        v_next_in_subrect = sub_rect.has_point(v_(i+1))
        if v_i_in_subrect and v_next_in_subrect:
            clipped.append(v_i)
        elif v_i_in_subrect and not v_next_in_subrect:
            clipped.append(v_i)
            clipped.append(intersect(v_i, v_next, sub_rect))
        elif not v_i_in_subrect and v_next_in_subrect:
            clipped.append(intersect(v_i, v_next, sub_rect))
        # else: both outside → skip both vertices
    if clipped.size() >= 3:
        # rescale clipped vertices from sub_rect-local coords back to tile-center-local
        # for the synthesized tile_size×tile_size output
        for v in clipped: v = (v - sub_rect.center) * (tile_size / sub_rect.size)
        synthesized_polygons.append(clipped)
```

`intersect(a, b, rect)` clips line segment `a→b` against rect's four edges using parametric line clipping (Liang-Barsky-style; ~30 LOC). Hide in `penta_tile_synthesis.gd` as `_clip_segment_to_rect`.

### Edge cases (LOCKED)

| Case | Disposition |
|---|---|
| Polygon crosses sub-region boundary | Clip per algorithm above; keep clipped fragment if ≥ 3 vertices remain. |
| Polygon entirely outside sub-region | Drop (no entry in synthesized tile). |
| Polygon entirely inside sub-region | Keep all vertices, just rescale. |
| Multiple occlusion polygons (`polygon_index > 0`) | Iterate via `TileData.get_occluder_polygons_count()` per layer; clip each independently; preserve `polygon_index` ordering on output. |
| Navigation polygons with hole loops | Preserve outer + hole loops; clip each loop independently; if outer loop drops, drop the entire navigation polygon (a hole without an outer is invalid). |
| Source polygon has < 3 vertices after clipping | Drop. |
| OuterCorner archetype (no synthesized output cell — slot 0 used verbatim with rotation) | Polygons NOT clipped/transformed at synthesis time. At render time, Godot applies the rotation transform to slot 0's polygons via TileSetAtlasSource's standard `TRANSFORM_FLIP_*` handling. |

### What is NOT copied (LOCKED — per PENTA-SYNTH-06)

The synthesizer copies ONLY these polygon types: collision (physics), occlusion (light), navigation. The following are explicitly NOT copied to synthesized tiles, documented as the layout-choice tradeoff per PENTA-SYNTH-06:

- Animation frames (`TileData.get_animation_frames_count`, `set_animation_frame_*`)
- Custom data layers (`TileData.get_custom_data*`)
- Probability weights (`TileData.probability`)
- Y-sort origin (`TileData.y_sort_origin`)
- Texture origin / modulate (`TileData.texture_origin`, `TileData.modulate`)
- Material override (`TileData.material`)
- Z-index (`TileData.z_index`)

Use a non-Penta layout (DualGrid16 / Wang2Edge / Wang2Corner / Min3x3) if any of the above metadata is required.

    1. Read tile_size from source_tile_set's source `source_id`.
    2. Run `validate_tile_size`; if any "REQUIRED square" or "REQUIRED even" warning fires, return Dictionary with warnings + empty slots.
    3. For each output slot 0..mode-1:
       - If slot index is "authored" for this mode (per the Gate 1 mode→archetype table), extract that source tile's image + polygons directly.
       - Else (synthesized archetype), call `_synthesize_archetype(source_tile_set, source_id, slot_0_coords, archetype_slot)` which:
         - Looks up sub-rect via `_subrect_for_archetype`
         - Extracts pixel sub-image: `source_image.get_region(subrect_in_pixels)` then `image.resize(tile_size.x, tile_size.y, Image.INTERPOLATE_NEAREST)` (NEAREST is mandatory — pixel-art fidelity; verified by acceptance criterion grep below)
         - Iterates polygon types (collision physics layer 0/1/.., occlusion, navigation), clips each via `clip_polygon_to_subrect`, accumulates output polygons keyed by (layer_type, polygon_index).
         - Returns `{ "atlas_coords": Vector2i, "image": Image, "polygons": Dictionary }`.
    4. Determinism: identical inputs → identical Dictionary output (same loop order, same sub-rect lookup, same Image data hash).
